# Limitations & Roadmap

## How Verification Works Today

```
QR generated (₹499.37) → Customer scans → Pays via UPI app
  → Bank sends email alert to merchant's Gmail
  → upiagent reads email, LLM extracts: amount, UPI ref, sender
  → Security pipeline: format ✓ → amount match ✓ → time window ✓ → dedup ✓
  → Payment marked as verified
```

## The Matching Problem

**How does upiagent know which payment a bank email belongs to?**

Currently: **amount matching within a time window.**

Each QR gets a unique amount via `addPaisa` (₹499 → ₹499.37). When the bank email arrives saying "₹499.37 credited", we match it to the pending payment with that exact amount.

### Where this breaks

| Scenario | Problem |
|----------|---------|
| 100+ concurrent payments at same base amount | Only 99 unique paisa values (0.01-0.99). Collisions likely. |
| Customer pays wrong amount | No match. Payment stays pending. |
| Bank email delayed > 5 minutes | Falls outside time window. Misses verification. |
| Two customers pay identical amounts at same time | Can't distinguish which email belongs to which payment. |
| LLM extracts wrong amount | False match or no match. |

### Why we can't use UPI Reference ID for matching

The UPI reference ID is unique per transaction — perfect for dedup. But we **can't predict it in advance**. It's generated by the banking system when the payment happens, not by us. So we can't pre-associate a UPI ref with a specific pending payment.

If we could query "which payments were made to VPA X in the last 5 minutes", we'd solve this. But UPI doesn't offer this API to non-bank entities.

## Current Capacity

With `addPaisa: true` and a 5-minute time window:

| Base Amount | Max Concurrent Payments | Collision Risk |
|-------------|------------------------|----------------|
| Any single amount | ~99 | Near zero for < 50 |
| Mixed amounts (different products) | ~99 per amount | Very low |
| Single product, high traffic | 99 | Collisions after ~70 (birthday paradox) |

**Safe for:** Small-medium merchants. Up to ~50 concurrent payments per price point.

**Not safe for:** Flash sales, ticketing platforms, high-frequency single-price products.

## Roadmap: Scaling Verification

### Phase 1: Current (MVP)
- Amount matching with addPaisa
- Gmail bank alert parsing
- 5-minute time window
- **Capacity: ~50 concurrent per price point**

### Phase 2: Multi-signal matching
- Match on amount + sender UPI ID (if customer provides it)
- Match on amount + partial timing correlation
- Screenshot upload as second verification source
- **Capacity: ~200 concurrent per price point**

### Phase 3: Virtual accounts (planned)
- Partner with a banking-as-a-service provider (e.g., Cashfree, Decentro)
- Generate unique virtual account numbers per transaction
- Payment to virtual account = guaranteed 1:1 match
- **Capacity: Unlimited**

### Phase 4: UPI Collect / Intent (planned)
- Become a registered Payment Aggregator with RBI
- Use UPI Collect API — send payment request directly to customer's UPI
- Real-time payment confirmation via callback
- **Capacity: Unlimited, instant verification**

## Mitigations (Available Now)

### 1. Use addPaisa (default: on)
Each QR gets a unique amount. ₹499 → ₹499.37. This is the primary disambiguation.

### 2. Short time windows
Default 5 minutes. Shorter = fewer collisions but more missed verifications if email is delayed.

### 3. Require customer info
If your checkout collects the customer's UPI ID or phone number, pass it to `verifyPayment({ expectedFrom: "customer@upi" })` for additional matching.

### 4. Sequential processing
Process one email → match to oldest pending payment with that amount → mark verified → next email. FIFO ordering prevents double-matching.

### 5. Manual fallback
For edge cases, merchants can manually verify payments in the dashboard using the UPI reference ID from the customer's screenshot.

## Honest Assessment

upiagent is **not a replacement for Razorpay at scale**. It's a replacement for Razorpay **for small merchants** who:
- Don't want to pay 2% per transaction
- Don't need 10K concurrent payments
- Want a self-hosted/open-source option
- Are building MVPs or side projects
- Need UPI payments without merchant onboarding

For high-traffic production use, the virtual account integration (Phase 3) or UPI Collect (Phase 4) is the correct solution. We're building toward that.
